# Envoy AI Gateway v0.4 Baseline Plan

이 문서는 `kakao-envoy` 프로젝트에서 Envoy AI Gateway v0.4를 greenfield baseline으로 구성하기 위한 실행 계획입니다.
목표는 v0.4 환경을 먼저 재현 가능하게 만든 뒤, 같은 목적의 구성을 v0.5 target으로 전환하는 마이그레이션 흐름을 검증하는 것입니다.

## 목적

이 프로젝트는 기존 운영 v0.4 매니페스트를 받아서 변환하는 작업이 아닙니다.
우리가 직접 v0.4 출발점을 만들고, 그 출발점을 v0.5 구조로 업그레이드하는 과정을 PoC 산출물로 남깁니다.

v0.4 baseline의 역할:

- v0.5 migration 이전의 출발점 제공
- v0.4 스타일 Gateway, Backend, AIServiceBackend, AIGatewayRoute 구성 확인
- v0.5에서 바뀌는 설정 차이를 비교할 기준 제공
- Memory ExtProc PoC가 올라가기 전 기본 LLM gateway 흐름 확인

## 기준 버전

| 구성요소 | v0.4 baseline 기준 |
|----------|--------------------|
| Envoy AI Gateway | v0.4.0 |
| Envoy Gateway | v1.5.x 이상 |
| Envoy Proxy | v1.35+ 계열 |
| Gateway API | v1.4.0 |
| Kubernetes | v1.32+ 권장 |

주의:

- v0.4 공식 설치 문서는 AI Gateway CRD chart와 controller chart를 분리해서 설치합니다.
- v0.5 target에서는 `GatewayConfig`를 도입해 ExtProc 설정과 환경변수를 관리합니다.

## 기본 전략

v0.4 baseline은 두 단계로 나눕니다.

### 1. Smoke baseline

공식 v0.4 `examples/basic/basic.yaml`을 사용해 API key 없이 기본 Gateway 동작을 먼저 확인합니다.

이 단계의 목적:

- Envoy Gateway와 Envoy AI Gateway v0.4 설치가 정상인지 확인
- Gateway pod 생성과 라우팅이 동작하는지 확인
- `curl` 요청에 응답이 돌아오는지 확인

이 단계에서는 Redis나 Memory ExtProc를 붙이지 않습니다.

### 2. Migration baseline

저장소의 v0.4 baseline 매니페스트를 사용해 v0.5 target과 비교할 전환 대상을 명시합니다.

파일:

- `deploy/gateway/v0.4-sample.yaml`

이 단계의 목적:

- `schema.version`에서 `schema.prefix`로 전환할 대상을 보여줍니다.
- `filterConfig.externalProcessor.resources`에서 `GatewayConfig.spec.extProc.resources`로 이동할 대상을 보여줍니다.
- v0.5 target 매니페스트와 비교 가능한 기준을 제공합니다.

## 설치 순서

### 1. Kind 클러스터 준비

```bash
kind create cluster --config deploy/kind/v0.4-cluster.yaml
```

예상 결과:

```text
Created cluster "ai-gateway-v04"
```

### 2. Envoy Gateway 설치

v0.4 baseline에서는 Envoy Gateway v1.5.x 이상을 사용합니다.
반복 실행을 줄이기 위해 저장소 스크립트를 우선 사용합니다.

```bash
deploy/gateway/v0.4/install.sh
```

스크립트가 실행하는 핵심 명령은 다음과 같습니다.

```bash
helm upgrade -i eg oci://docker.io/envoyproxy/gateway-helm \
  --version v1.5.0 \
  --namespace envoy-gateway-system \
  --create-namespace \
  -f https://raw.githubusercontent.com/envoyproxy/ai-gateway/v0.4.0/manifests/envoy-gateway-config/config.yaml

kubectl wait --timeout=2m \
  -n envoy-gateway-system \
  deployment/envoy-gateway \
  --for=condition=Available
```

주의:

- 실제 v0.4 설치 검증 시 공식 문서와 chart compatibility를 다시 확인합니다.
- Envoy Gateway values 파일 경로가 변경되었거나 동작하지 않으면 이 내용을 `docs/issues/`에 기록합니다.

### 3. Envoy AI Gateway v0.4 CRD 설치

```bash
helm upgrade -i aieg-crd oci://docker.io/envoyproxy/ai-gateway-crds-helm \
  --version v0.4.0 \
  --namespace envoy-ai-gateway-system \
  --create-namespace
```

### 4. Envoy AI Gateway v0.4 controller 설치

```bash
helm upgrade -i aieg oci://docker.io/envoyproxy/ai-gateway-helm \
  --version v0.4.0 \
  --namespace envoy-ai-gateway-system \
  --create-namespace

kubectl wait --timeout=2m \
  -n envoy-ai-gateway-system \
  deployment/ai-gateway-controller \
  --for=condition=Available
```

### 5. 설치 확인

```bash
kubectl get pods -n envoy-gateway-system
kubectl get pods -n envoy-ai-gateway-system
kubectl get crd | grep aigateway
```

예상 결과:

- Envoy Gateway pod가 `Running` 또는 `Available` 상태여야 합니다.
- AI Gateway controller pod가 `Running` 또는 `Available` 상태여야 합니다.
- `aigateway.envoyproxy.io` CRD가 보여야 합니다.

## Smoke baseline 검증

### 1. 공식 basic 예제 적용

저장소 스크립트를 사용하면 예제 적용, Gateway pod 대기, port-forward, curl 검증을 한 번에 실행합니다.

```bash
deploy/gateway/v0.4/smoke-test.sh
```

스크립트가 실행하는 핵심 흐름은 다음과 같습니다.

```bash
kubectl apply -f https://raw.githubusercontent.com/envoyproxy/ai-gateway/v0.4.0/examples/basic/basic.yaml
```

### 2. Gateway pod 준비 대기

```bash
kubectl wait pods --timeout=2m \
  -l gateway.envoyproxy.io/owning-gateway-name=envoy-ai-gateway-basic \
  -n envoy-gateway-system \
  --for=condition=Ready
```

### 3. Gateway URL 설정

로컬 Kind 환경에서는 LoadBalancer external IP가 없을 수 있으므로 port-forward를 기본 검증 경로로 사용합니다.

```bash
kubectl port-forward -n envoy-gateway-system svc/eg-envoy-ai-gateway 8080:80
```

다른 터미널에서:

```bash
export GATEWAY_URL=http://localhost:8080
```

주의:

- 실제 service 이름은 클러스터에서 생성된 이름을 확인해야 합니다.
- 확인 명령:

```bash
kubectl get svc -n envoy-gateway-system
```

### 4. Chat Completions 요청

```bash
curl -H "Content-Type: application/json" \
  -d '{
        "model": "some-cool-self-hosted-model",
        "messages": [
          {
            "role": "system",
            "content": "Hi."
          }
        ]
      }' \
  "$GATEWAY_URL/v1/chat/completions"
```

예상 결과:

```json
{
  "choices": [
    {
      "message": {
        "content": "I'll be back."
      }
    }
  ]
}
```

## Migration baseline 검증

저장소의 baseline 매니페스트는 migration diff를 보여주기 위한 기준입니다.
먼저 YAML 문법을 확인합니다.

```bash
ruby -e 'require "yaml"; ARGV.each { |f| YAML.load_stream(File.read(f)); puts "ok #{f}" }' \
  deploy/gateway/v0.4-sample.yaml
```

클러스터 CRD가 준비된 뒤 server dry-run을 실행합니다.

```bash
kubectl apply --dry-run=server -f deploy/gateway/v0.4-sample.yaml
```

완료 기준:

- v0.4 baseline 매니페스트가 CRD schema 기준으로 유효합니다.
- baseline에 `schema.version`이 존재합니다.
- baseline에 `filterConfig.externalProcessor.resources`가 존재합니다.
- v0.5 target과 비교할 전환 대상이 명확합니다.

## v0.5 전환으로 이어지는 다음 단계

v0.4 baseline이 준비되면 다음 순서로 v0.5 target을 검증합니다.

1. v0.5 CRD와 controller 설치
2. `deploy/gateway/v0.5-gateway-config-sample.yaml` server dry-run
3. `schema.version` 제거와 `schema.prefix` 적용 확인
4. `filterConfig.externalProcessor.resources` 제거와 `GatewayConfig.spec.extProc.resources` 적용 확인
5. Redis와 Memory ExtProc 배포
6. 세션 메모리 요청 시나리오 검증

## 의사결정 필요 후보

아래 항목은 실제 클러스터 검증 전에 사용자가 선택해야 할 수 있습니다.

| 항목 | 기본 제안 | 이유 |
|------|-----------|------|
| v0.4 smoke test 방식 | 공식 basic 예제 사용 | API key 없이 빠르게 Gateway 동작 확인 가능 |
| 실제 provider 연동 시점 | v0.5 target 이후 | 메모리 PoC 검증과 provider credential 관리를 분리 |
| Kind 클러스터 이름 | `ai-gateway-v04` | v0.5 검증 클러스터와 분리 가능 |
| Redis 포함 시점 | v0.5 memory 단계 | v0.4 baseline은 migration 출발점 역할에 집중 |
