# 프로젝트 목적 쉽게 이해하기

이 문서는 `kakao-envoy` 프로젝트를 왜 하는지 비유로 쉽게 설명하기 위한 문서입니다.
기술 구현의 기준은 `README.md`와 `docs/architecture.md`를 따르되, 프로젝트의 큰 그림을 빠르게 공유할 때는 이 문서를 참고합니다.

## 한 문장 요약

회사의 여러 서비스가 LLM을 제각각 직접 붙이지 않고, 공통 AI Gateway를 통해 안전하고 일관되게 사용하게 만들고, 그 Gateway 주변에 Redis 기반 대화 메모리 기능을 붙일 수 있는지 검증하는 프로젝트입니다.

더 쉽게 말하면:

```text
회사 AI 사용을 위한 공용 정문을 만들고,
그 정문 옆에 사용자별 대화 메모장을 챙겨주는 비서를 붙이는 실험입니다.
```

## 회사 건물 출입구 비유

회사의 여러 서비스가 각각 LLM을 쓰고 싶어 한다고 가정합니다.

예를 들면:

- 고객 상담 챗봇
- 내부 문서 검색 Agent
- 코드 리뷰 Agent
- 업무 자동화 Agent
- 카카오톡, 웹, 앱 기반 AI 기능

각 서비스가 모델 API를 직접 호출하면 구조가 이렇게 됩니다.

```text
서비스 A -> OpenAI
서비스 B -> Gemini
서비스 C -> 사내 모델
서비스 D -> 또 다른 LLM
```

이 방식은 처음에는 단순해 보이지만, 회사 전체 관점에서는 관리가 어려워집니다.

- 누가 어떤 모델을 얼마나 쓰는지 알기 어렵습니다.
- 인증과 권한을 서비스마다 따로 만들어야 합니다.
- 요청 제한, 로깅, 비용 관리 기준이 흩어집니다.
- 모델 교체나 라우팅 정책을 중앙에서 바꾸기 어렵습니다.
- 공통 정책을 적용하기 어렵습니다.

그래서 회사 AI 사용의 공용 출입구로 **AI Gateway**를 둡니다.

```text
서비스 A
서비스 B
서비스 C
서비스 D
   ↓
AI Gateway
   ↓
OpenAI / Gemini / 사내 LLM / 기타 모델
```

AI Gateway는 말하자면 **회사 AI 사용의 정문**입니다.
각 서비스는 모델 회사로 직접 뛰어가지 않고, 이 정문을 통해 LLM을 호출합니다.

## AI Gateway가 맡는 공통 역할

정문에서 회사는 공통 기능을 처리할 수 있습니다.

```text
인증: 이 서비스가 이 모델을 써도 되는가?
라우팅: 이 요청은 OpenAI로 보낼까, Gemini로 보낼까?
제한: 이 서비스는 분당 몇 번까지 호출할 수 있는가?
로깅: 누가 언제 어떤 모델을 썼는가?
비용 관리: 팀별, 서비스별 사용량은 얼마나 되는가?
정책 관리: 민감한 요청은 막거나 다른 모델로 보낼 것인가?
```

이것이 Envoy AI Gateway를 쓰는 큰 이유입니다.

## 이 프로젝트에서 추가로 검증하는 것: 대화 메모리

이 프로젝트에는 AI Gateway 위에 하나의 목표가 더 있습니다.
바로 **대화 메모리**입니다.

LLM은 기본적으로 이전 요청을 자동으로 기억하지 않습니다.
매 요청은 독립적으로 처리되기 때문에, 이전 대화를 다시 넣어주지 않으면 사용자가 무슨 말을 했는지 모를 수 있습니다.

예를 들어 첫 번째 요청에서 사용자가 말합니다.

```text
사용자: 내 이름은 유리야.
LLM: 반가워, 유리야.
```

두 번째 요청에서 사용자가 묻습니다.

```text
사용자: 내 이름이 뭐라고 했지?
```

이전 대화를 같이 보내주지 않으면 LLM은 답하지 못할 수 있습니다.

그래서 보통 애플리케이션이 이전 대화를 저장해두고, 다음 요청 때 함께 보내야 합니다.

```text
예전 대화:
사용자: 내 이름은 유리야.
AI: 반가워, 유리야.

이번 질문:
사용자: 내 이름이 뭐라고 했지?
```

그러면 LLM은 이전 문맥을 보고 답할 수 있습니다.

```text
AI: 유리라고 하셨어요.
```

## 우리 프로젝트의 비유

이 프로젝트의 구조를 비유하면 다음과 같습니다.

```text
AI Gateway = 회사 AI 정문
Memory ExtProc = 정문 옆 비서
Redis = 비서가 들고 있는 메모장
```

서비스가 Gateway로 요청을 보내면, 정문 옆 비서인 `Memory ExtProc`가 먼저 확인합니다.

```text
비서: 이 사용자의 session id가 뭐지?
비서: Redis 메모장에 예전 대화가 있나?
비서: 있네. 그럼 이번 질문 앞에 예전 대화를 붙여서 보내자.
```

그래서 실제 LLM에게는 다음처럼 요청이 전달됩니다.

```text
예전 대화:
사용자: 내 이름은 유리야.
AI: 반가워, 유리야.

이번 질문:
사용자: 내 이름이 뭐라고 했지?
```

LLM이 답하면, 비서가 다시 메모장에 저장합니다.

```text
Redis:
- 사용자: 내 이름은 유리야.
- AI: 반가워, 유리야.
- 사용자: 내 이름이 뭐라고 했지?
- AI: 유리라고 하셨어요.
```

## 실제 처리 흐름

```text
서비스
  ↓
Envoy AI Gateway
  ↓
Memory ExtProc
  ↓
Redis에서 이전 대화 조회
  ↓
messages에 히스토리 주입
  ↓
LLM 호출
  ↓
응답 저장
```

## v0.5로 전환하는 이유

중요한 점은 **Envoy AI Gateway v0.5가 메모리를 기본 제공해서 전환하는 것이 아니라는 것**입니다.

v0.5로 전환하는 이유는 다음 구조를 붙이기 좋아졌기 때문입니다.

```text
GatewayConfig
= Gateway 주변 설정을 더 명확하게 관리

EnvoyExtensionPolicy
= 우리가 만든 Memory ExtProc를 Gateway에 연결

Body Mutation / ExtProc 흐름
= 요청 내용을 가공해서 LLM에 보내기 좋음
```

즉 이 프로젝트의 메모리 기능은 Redis 하나가 아니라 다음 조합으로 만들어집니다.

```text
Memory Feature = Envoy AI Gateway v0.5 확장 구조 + Custom ExtProc + Redis
```

## 정리

이 프로젝트는 단순히 LLM 호출 예제를 만드는 작업이 아닙니다.

목표는 회사 내부 여러 서비스가 LLM을 공통 Gateway를 통해 사용하게 만들고,
Gateway 주변 계층에서 공통 정책과 대화 메모리 기능을 관리할 수 있는지 검증하는 것입니다.

따라서 각 구성 요소의 역할은 다음과 같습니다.

| 구성 요소 | 쉬운 비유 | 실제 역할 |
|----------|-----------|-----------|
| Envoy AI Gateway | 회사 AI 정문 | LLM 요청/응답이 지나가는 공통 Gateway |
| Memory ExtProc | 정문 옆 비서 | Redis 히스토리를 조회하고 `messages`에 주입 |
| Redis | 비서의 메모장 | 세션별 대화 히스토리 저장 |
| GatewayConfig | 정문 운영 설정 | AI Gateway ExtProc 환경변수와 리소스 관리 |
| EnvoyExtensionPolicy | 비서 연결 규칙 | Custom ExtProc 서비스를 Gateway에 연결 |
