# Issue and Incident Log Guide

이 문서는 프로젝트 진행 중 발생하는 문제, 특이 사항, 의사결정 변경, 검증 실패를 기록하는 기준입니다.
목표는 문제가 생겼을 때 원인과 해결 과정을 나중에 재현할 수 있게 남기는 것입니다.

## 기록 대상

다음 중 하나에 해당하면 기록합니다.

- 명령어, 테스트, 배포, dry-run, 빌드가 실패한 경우
- 공식 문서와 기존 README/AGENTS 내용이 서로 다르거나 충돌한 경우
- 버전, CRD, API 필드, Helm chart 값처럼 프로젝트 방향에 영향을 주는 차이를 발견한 경우
- 사용자의 결정으로 기존 Seed 방향이 바뀐 경우
- 임시 우회, fallback, TODO, 리스크를 남긴 경우
- 같은 문제가 다시 발생할 가능성이 있는 경우

사소한 오타 수정이나 즉시 해결된 단순 편집 실수는 남기지 않아도 됩니다.

## 저장 위치

문제/특이 사항 기록은 `docs/issues/` 아래에 Markdown 파일로 저장합니다.

파일명 형식:

```text
YYYY-MM-DD-short-title.md
```

예:

```text
docs/issues/2026-04-23-gatewayconfig-api-version.md
```

## 작성 시점

가능하면 문제가 확인된 같은 작업 Seed 안에서 작성합니다.

권장 흐름:

1. 문제 발견
2. 영향 범위 확인
3. 원인 또는 추정 원인 정리
4. 해결 또는 임시 대응
5. 검증 결과 기록
6. 남은 리스크와 후속 작업 기록

해결 전이라도 중요한 문제라면 먼저 기록하고, 해결 후 같은 문서에 업데이트합니다.

## 템플릿

````markdown
# <문제 요약>

## 상태

- 상태: open | resolved | deferred
- 발견일: YYYY-MM-DD
- 해결일: YYYY-MM-DD 또는 N/A
- 관련 Seed: Seed N - <이름>

## 배경

문제가 발견된 맥락을 적습니다.

## 증상

실패한 명령어, 오류 메시지, 문서 충돌, 이상 동작을 적습니다.

```text
<필요한 경우 오류 출력 일부>
```

## 영향

- 어떤 파일, 설정, 테스트, 배포 흐름에 영향을 주는지 적습니다.
- 사용자 목표나 프로젝트 일정에 영향이 있으면 적습니다.

## 원인

확인된 원인 또는 현재 가장 가능성 높은 추정을 적습니다.
추정이면 추정이라고 표시합니다.

## 해결

적용한 변경, 우회, 결정 사항을 적습니다.

## 검증

실행한 검증과 결과를 적습니다.

```bash
<검증 명령어>
```

예상 또는 실제 결과:

```text
<결과>
```

## 남은 리스크

- 아직 검증하지 못한 부분
- 실제 클러스터나 외부 환경이 필요해 남은 부분
- 재발 가능성

## 관련 파일

- `path/to/file`

## 참고 자료

- [문서 제목](https://example.com)
````

## 작성 원칙

- 오류 출력은 필요한 부분만 발췌하고, API key나 token 같은 민감정보는 남기지 않습니다.
- 해결 과정에서 사용한 명령어와 결과를 구분해 적습니다.
- 확인된 사실과 추정을 분리합니다.
- 문제가 해결되면 상태를 `resolved`로 바꾸고 해결일을 채웁니다.
- 후속 작업이 필요하면 다음 Seed 후보로 연결합니다.
